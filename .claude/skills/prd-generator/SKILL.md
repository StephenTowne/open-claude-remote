# PRD Generator

## Overview

Generate comprehensive, well-structured Product Requirements Documents (PRDs) that serve two audiences:
1. **Human stakeholders** — product managers, engineers, designers, and leadership
2. **Downstream AI agents** — architecture design, system design, and UI/UX design agents that consume the PRD as structured input

Every PRD produced by this skill ends with an **AI Agent Briefs** section containing three machine-readable subsections for downstream AI consumption.

## Core Workflow

When a user requests to create a PRD, follow this workflow:

### Step 1: Gather Context

> ⚠️ **MANDATORY GATE — Do NOT read template files, do NOT generate any document content, and do NOT proceed to Step 2 until this step is fully complete.**

Count how many of the 6 required fields the user has explicitly provided. If **fewer than 5 of 6** are answered, you MUST ask the discovery questions and wait for the user's responses before doing anything else.

**Required Information (6 fields):**
- **Feature/Product Name**: What are we building?
- **Problem Statement**: What problem does this solve?
- **Target Users**: Who is this for?
- **Business Goals**: What are we trying to achieve?
- **Success Metrics**: How will we measure success?
- **Timeline/Constraints**: Any deadlines or limitations?

**Discovery Questions (ask all that are unanswered):**
```
1. What problem are you trying to solve?
2. Who is the primary user/audience for this feature?
3. What are the key business objectives?
4. Are there any technical constraints we should be aware of?
5. What does success look like? How will you measure it?
6. What's the timeline for this feature?
7. What's explicitly out of scope?
```

> Only skip a question if the user's input **explicitly and unambiguously** answers it. A one-sentence description does NOT count as sufficient — it typically covers at most 1–2 fields. When in doubt, ask.

### Step 2: Generate PRD

Use `references/prd_template.md` to create the document. The PRD covers:

1. Executive Summary
2. Problem Statement
3. Goals & Objectives
4. User Personas
5. User Stories & Requirements (with acceptance criteria)
6. Success Metrics
7. Scope (in / out)
8. Technical Considerations
9. Design & UX Requirements
10. Timeline & Milestones
11. Risks & Mitigation
12. Dependencies & Assumptions
13. Open Questions
14. **AI Agent Briefs** ← always include; see section below

Reference `references/user_story_examples.md` for user story patterns and `references/metrics_frameworks.md` for metrics guidance (AARRR, HEART, OKRs).

### Step 3: Define Success Metrics

Choose the appropriate framework based on product type:

- **AARRR**: Acquisition, Activation, Retention, Revenue, Referral
- **HEART**: Happiness, Engagement, Adoption, Retention, Task Success
- **North Star Metric**: Single key metric representing core value
- **OKRs**: Objectives and Key Results

### Step 4: Validate & Review

Run the validation script if available (Claude Code environment):

```bash
scripts/validate_prd.sh <prd_file.md>
# Verbose: scripts/validate_prd.sh <prd_file.md> --verbose
```

Checks for: required sections present, user story format, success metrics defined, scope articulated, no placeholder text, AI Agent Briefs populated.

**If the script is unavailable** (Claude.ai or script not found), manually verify every item in the Validation Checklist below before delivering the PRD. Do not skip validation entirely — the checklist is the fallback, not an optional step.

## AI Agent Output

Every PRD generated by this skill includes an **AI Agent Briefs** section at the end, designed for machine consumption by downstream AI agents. This section contains three structured subsections (defined in `references/prd_template.md`):

- **Architecture Design Brief** — functional scope, domain entities, integrations, auth model, non-functional requirements; consumed by architecture design agents
- **System Design Constraints** — data entities, API surface, performance budgets, scalability targets, deployment constraints; consumed by system design agents
- **UI/UX Design Requirements** — user flows, screen inventory, interaction patterns, accessibility requirements, design system references; consumed by UI/UX design agents

Always populate these sections when generating a PRD. They are the primary bridge between product requirements and technical/design execution in AI-driven workflows.

## Usage Examples

**New feature:** "Create a PRD for adding biometric authentication to our iOS app"
→ Asks about security requirements, existing auth, user personas → generates full PRD + AI Agent Briefs

**Enhancement:** "Write requirements for improving our checkout flow conversion"
→ Gathers conversion data, drop-off points → generates PRD with before/after metrics + AI Agent Briefs

**New product:** "I need a PRD for an admin dashboard for enterprise customers"
→ Identifies B2B requirements (RBAC, SSO, multi-tenancy) → generates comprehensive PRD + AI Agent Briefs

## Validation Checklist

Before finalizing, verify:

- [ ] Problem is clear and anyone can understand what we're solving
- [ ] Users are identified with personas
- [ ] Success metrics are measurable with clear targets
- [ ] Scope is bounded (explicit in/out-of-scope)
- [ ] Requirements are testable (QA can verify)
- [ ] **AI Agent Briefs section is fully populated** (not left as placeholders)

## Troubleshooting

### Insufficient user input
See the mandatory gate in Step 1: if fewer than 5 of the 6 required fields are explicitly provided, ask the discovery questions before generating. Never infer or invent answers to missing fields. Never produce a PRD with placeholder content such as `[TBD]` or `[Insert here]`.

### Reference files not found
If `references/prd_template.md` is not accessible, generate the PRD using the 14-section structure defined in Step 2 of the Core Workflow. If `references/user_story_examples.md` or `references/metrics_frameworks.md` are missing, apply standard As a / I want / So that user story format and default to HEART metrics for consumer products, OKRs for internal tools.

### Validation script unavailable or fails
If `scripts/validate_prd.sh` is not found or returns an error, fall back to the manual Validation Checklist. If the script exits non-zero, address each reported issue before delivering the PRD — do not deliver a PRD that fails validation.

### AI Agent Briefs left as placeholders
If any subsection of AI Agent Briefs (Architecture Design Brief, System Design Constraints, UI/UX Design Requirements) cannot be populated due to insufficient context, ask the user for the missing information rather than leaving placeholders. These sections are required for downstream AI workflows.

### PRD scope is unclear
If the user cannot define what is out of scope, suggest explicit exclusions based on common patterns for the feature type (e.g., for auth features: "Out of scope: social login, password recovery v2, admin user management"). Confirm with the user before including.
