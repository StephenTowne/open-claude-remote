# Requirement Criteria — 详细判定标准

Step 3 的完整判定参考。SKILL.md 中的二问流程为简要版，本文件为权威标准。

---

## Q1: 用户可见变更判定标准

满足**任意一项**即为 YES：

| 判定项 | 说明 | 示例 |
|--------|------|------|
| 新增功能点 | 新增产品功能 | 新增通知渠道、新增暗黑模式 |
| 功能范围扩展 | 从单一扩展到多个 | 单渠道 → 多渠道列表 |
| 用户可见行为变更 | 操作流程或 UI 变化 | 新增配置页面、交互方式改变 |
| 新增配置项 | 环境变量、配置字段 | 新增 `THEME` 变量 |
| 支持新场景 | 新的使用方式 | 支持移动端手势操作 |

**写入位置**: `docs/requirements/*.md`（功能需求文档）

---

## Q2: 架构模式变更判定标准

Q1 = NO 时进入此判定。满足**任意一项**即为 YES：

| 判定项 | 说明 | 示例 |
|--------|------|------|
| 数据流方向改变 | 同步↔异步、轮询↔推送、单向↔双向 | 从 HTTP 轮询改为 WebSocket 推送 |
| 模块分层调整 | 拆分/合并模块目录、引入新抽象层 | session 拆分为 session + instance |
| 通信协议变更 | 消息格式重构、新增消息类型族 | WebSocket 协议新增 instance 消息族 |
| 状态管理方式改变 | 本地↔集中式、引入新缓存策略 | 从组件本地状态改为全局 store |

**写入位置**: `docs/requirements/tech-*.md`（技术变更文档）

---

## 明确不算架构变更

以下情况 Q2 = NO：

| 排除项 | 说明 |
|--------|------|
| 提取函数/重命名 | 同模块内的代码整理 |
| 调整文件位置 | 同模块内移动文件 |
| 性能优化 | 加缓存但不改数据流方向 |
| Bug 修复 | 修复已有功能错误 |
| 补测试 | 测试文件的新增/修改 |
| 内部 API 变更 | 用户不直接接触的接口调整 |

---

## 边界案例

| 场景 | Q1 | Q2 | 结论 |
|------|----|----|------|
| 新增 WebSocket 消息类型（1-2 个，服务现有功能） | NO | NO | 无需文档 |
| 新增 WebSocket 消息类型**族**（全新消息类别） | NO | YES | tech-*.md |
| 加 Redis 缓存层（对外行为不变） | NO | YES | tech-*.md（引入新状态管理层） |
| 加内存缓存（Map 缓存函数结果） | NO | NO | 无需文档（性能优化） |
| 拆分大文件为多个小文件（同目录） | NO | NO | 无需文档（文件整理） |
| 拆分模块为独立子模块（新目录+新职责边界） | NO | YES | tech-*.md |
| 从 REST 改为 GraphQL | NO | YES | tech-*.md（通信协议变更） |
| 新增环境变量但无 UI 变化 | YES | — | requirements/*.md |

## tech-*.md 文件模板

文件名：`docs/requirements/tech-{kebab-case-名称}.md`

```markdown
# [技术变更标题]

## 变更描述
[做了什么、为什么这么做]

## 影响范围
- 影响模块: [list]
- 架构模式变更: [具体描述哪个模式从什么变成什么]

## 架构决策
[为什么选择这个方案而非其他]
```

---

## 命名惯例

- 功能需求: `docs/requirements/{feature-name}.md`（如 `dark-mode.md`）
- 技术变更: `docs/requirements/tech-{change-name}.md`（如 `tech-instance-isolation.md`）
- 查看 `docs/requirements/` 下现有文件确认命名风格
